## DATABASE SETUP - Инфраструктура БД (ЭТАП 2)

### 📋 Обзор

Реализована полная инфраструктура базы данных для FastAPI приложения с использованием:
- **FastAPI** - веб-фреймворк
- **PostgreSQL** - реляционная БД
- **SQLAlchemy 2.0 async** - ORM с асинхронной поддержкой
- **Alembic** - управление миграциями
- **Pydantic Settings** - управление конфигурацией

### 📄 Описание каждого файла и его назначение

#### **1. `app/core/config.py` - Управление конфигурацией**

**Назначение:** Централизованное управление параметрами приложения через переменные окружения.

**Что реализовано:**
- Класс `Settings` на основе `pydantic_settings.BaseSettings`
- Чтение переменных из файла `.env`
- Параметры подключения к БД: URL, размер пула соединений, переполнение пула
- Глобальный объект `settings` для доступа по всему приложению

**Параметры:**
- `DATABASE_URL` - строка подключения к PostgreSQL
- `DATABASE_POOL_SIZE` - размер пула соединений (20 по умолчанию)
- `DATABASE_MAX_OVERFLOW` - максимальное переполнение пула (10 по умолчанию)
- `ADMIN_EMAIL` - опциональный email администратора

**Почему это нужно:**
- ✅ Отделение конфигурации от кода
- ✅ Безопасность - секреты не в коде
- ✅ Разные конфигурации для разных окружений (dev, test, prod)
- ✅ Легко менять параметры без перезагрузки кода

---

#### **2. `app/core/database.py` - Инициализация БД**

**Назначение:** Создание асинхронного engine и sessionmaker для работы с БД.

**Что реализовано:**
- **`engine`** - асинхронный SQLAlchemy engine с конфигурацией пула
  - Использует `asyncpg` драйвер для работы с PostgreSQL
  - Настроены параметры пула для оптимальной производительности
  
- **`async_session_maker`** - фабрика сессий
  - Создаёт новые асинхронные сессии
  - `expire_on_commit=False` - объекты остаются валидны после коммита

- **`Base`** - декларативный базовый класс
  - Все модели наследуют от `Base`
  - Используется для автогенерации таблиц и миграций

**Архитектура:**
```
engine (connection pool) 
    ↓
async_session_maker (session factory)
    ↓
Session (для каждого запроса)
    ↓
Models (User, Properties, Contract, etc.)
```

**Почему это нужно:**
- ✅ Асинхронность - FastAPI может обрабатывать множество запросов параллельно
- ✅ Пул соединений - переиспользование соединений для производительности
- ✅ Centralized Base - единое место для управления моделями

---

#### **3. `app/core/dependencies.py` - Зависимости для роутов**

**Назначение:** Функция-зависимость для внедрения сессии БД в эндпоинты.

**Что реализовано:**
- Функция `get_db()` с декоратором `yield`
- Создаёт новую сессию для каждого запроса
- Автоматически закрывает сессию после обработки запроса

**Использование в роутах:**
```python
@router.get("/users")
async def get_users(db: AsyncSession = Depends(get_db)):
    # db - это асинхронная сессия
    users = await db.execute(select(User))
    return users.scalars().all()
```

**Почему это нужно:**
- ✅ FastAPI автоматически управляет жизненным циклом сессии
- ✅ Чистый код без дублирования
- ✅ Гарантированное закрытие ресурсов

---

#### **4. `app/models/__init__.py` - Базовые классы и экспорты**

**Назначение:** Центральное место для определения базовых типов и экспорта.

**Что реализовано:**
- **`BaseModel`** - абстрактный базовый класс для всех моделей
  - Содержит служебные поля `created_at` и `updated_at`
  - Автоматически устанавливаются БД при создании/обновлении записи
  
- **Импорты утилит** - экспортирует часто используемые типы и классы
  - `Base` - для наследования
  - `Mapped`, `mapped_column`, `relationship` - для определения полей
  - SQL типы: `String`, `Text`, `Numeric`, `ForeignKey`, `TIMESTAMP`
  - Python типы: `datetime`, `Decimal`, `Optional`

- **TYPE_CHECKING** - импорты моделей только для type hints
  - Избегает циклических импортов
  - Сохраняет правильность типизации

**Почему это нужно:**
- ✅ DRY принцип - общие поля в одном месте
- ✅ Облегчает добавление новых моделей
- ✅ Избегает циклических импортов
- ✅ Улучшает читаемость

#### **5. Модели данных (user.py, property.py, contract.py, request.py, payment.py)**

**Общая структура:**
```python
class ModelName(BaseModel):
    __tablename__ = 'table_name'
    
    # Первичный ключ
    id: Mapped[int] = mapped_column(primary_key=True, autoincrement=True)
    
    # Обычные поля
    name: Mapped[str] = mapped_column(String(255), nullable=False)
    
    # Внешние ключи
    owner_id: Mapped[int] = mapped_column(ForeignKey('users.id'), nullable=False)
    
    # Связи с другими моделями
    owner: Mapped['User'] = relationship('User', back_populates='properties')
```

**5.1 User (user.py) - Пользователи**
- `id` - первичный ключ
- `name` - имя пользователя
- `email` - электронная почта (уникальная, с индексом)
- `password_hash` - хэш пароля
- Связи: владелец недвижимости (`properties`), арендатор (`contracts`)

**5.2 Properties (property.py) - Объекты недвижимости**
- `id` - первичный ключ
- `owner_id` - внешний ключ на пользователя (владельца)
- `title` - название (например, "Студия в центре")
- `address` - адрес
- `description` - описание (опционально)
- `property_type` - тип (квартира, дом, офис и т.д.)
- Связи: владелец (`owner`), договоры (`contracts`)

**5.3 Contract (contract.py) - Договоры аренды**
- `id` - первичный ключ
- `property_id` - внешний ключ на объект
- `tenant_id` - внешний ключ на арендатора (пользователя)
- `start_date` - дата начала
- `end_date` - дата окончания
- `status` - статус (активный, завершён и т.д.)
- `monthly_payment` - ежемесячный платёж (Decimal для денег)
- Связи: объект (`property`), арендатор (`tenant`), заявки (`requests`), платежи (`payments`)

**5.4 Request (request.py) - Заявки на обслуживание**
- `id` - первичный ключ
- `contract_id` - внешний ключ на договор
- `property_id` - внешний ключ на объект
- `message` - текст заявки
- `status` - статус (новая, в работе, выполнена)
- `request_date` - дата подачи
- Связи: договор (`contract`), объект (`property`)

**5.5 Payment (payment.py) - Платежи**
- `id` - первичный ключ
- `contract_id` - внешний ключ на договор
- `amount` - сумма платежа
- `due_date` - дата оплаты
- `paid_at` - дата фактической оплаты (опционально, NULL если не оплачено)
- `status` - статус (ожидается, оплачено, просрочено)
- `comment` - комментарий (опционально)
- Связь: договор (`contract`)

**Почему это нужно:**
- ✅ Строгая типизация через SQLAlchemy ORM
- ✅ Автоматическое создание таблиц и связей
- ✅ Миграции через Alembic
- ✅ Использование Decimal для денег (точность)
- ✅ Временные зоны в датах (timezone=True)

### 🔄 Поток данных

```
1. HTTP запрос к API
    ↓
2. FastAPI маршрутизирует к обработчику
    ↓
3. get_db() зависимость создаёт сессию
    ↓
4. async_session_maker() создаёт AsyncSession
    ↓
5. engine берёт соединение из пула
    ↓
6. Обработчик работает с моделями через сессию
    ↓
7. yield завершает зависимость
    ↓
8. Сессия закрывается, соединение возвращается в пул
    ↓
9. FastAPI отправляет ответ клиенту
```
